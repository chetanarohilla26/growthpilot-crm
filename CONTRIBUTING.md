# Documentation Standards

Documentation should be:

- Clear
- Professional
- Business-oriented
- Easy to understand
- Free from unnecessary jargon
- Well structured
- Consistent with existing formatting

Use Markdown formatting consistently.

Prefer:

- Headings
- Tables
- Bullet lists
- Diagrams
- Flowcharts

over long paragraphs.

---

When documenting workflows, include:

- Objective
- Inputs
- Outputs
- Owner
- Dependencies
- Process Steps
- Success Metrics
- Risks
- Automation Opportunities

- Lifecycle Stages
- Pipeline Stages
- Contact Properties

# AI Documentation

When documenting AI workflows, include:

- Purpose
- Recommended Prompt
- Expected Output
- Human Validation Required
- Limitations
- Best Practices

Never assume AI output is automatically correct.

Human review should always be part of the workflow.

# Pull Request Process

1. Fork the repository.

2. Create a feature branch.

```bash
git checkout -b feature/improve-documentation
```

3. Make your changes.

4. Commit your work.

5. Push your branch.

```bash
git push origin feature/improve-documentation
```

6. Open a Pull Request.

Please include:

- Summary
- Motivation
- Screenshots (if applicable)
- Files changed

---

# Commit Message Guidelines

Use descriptive commit messages.

Examples:

```
docs: improve CRM workflow documentation

docs: update customer lifecycle

feat: add lead routing template

fix: correct pipeline diagram

docs: improve README examples
```

Avoid messages like:

```
update

changes

fix

new file
```

# Reporting Issues

When creating an Issue, include:

## Description

Clearly describe the problem.

---

## Expected Behaviour

Explain what should happen.

---

## Current Behaviour

Explain what currently happens.

---

## Suggested Solution

Provide recommendations if available.

---

## Screenshots

Attach screenshots where relevant.

# Branch Naming

Recommended branch names:

```
feature/new-dashboard

feature/crm-workflow

feature/new-template

feature/automation

docs/readme-update

docs/improve-sop

fix/typo

fix/workflow
```

---

# File Naming

Use:

```
lowercase

hyphens

descriptive names
```

Example:

```
lead-routing.md

customer-lifecycle.md

crm-dashboard.md
```

Avoid:

```
New File.md

Document1.md

Final Version.md
```
